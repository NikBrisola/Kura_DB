01 · Visão geral

O que é o banco e como funciona.

KURA usa um banco Oracle 19c compartilhado entre dois backends. O banco é único, mas cada serviço tem responsabilidade clara sobre quais tabelas pode escrever. Isso evita conflito, duplicação de dado e bugs de integração.

Princípio central

O banco não tem "dono" — mas cada tabela tem. Felipe (.NET) nunca escreve em CONTA_TUTOR. Nikolas (Java) nunca escreve em EVENTO_CLINICO. Quem lê qualquer tabela? Ambos, quando necessário. Essa regra evita 90% dos bugs de integração antes mesmo de acontecerem.

02 · Ownership

Quem escreve em qual tabela.

A regra é simples: se você não está na coluna "escreve" de uma tabela, você só pode ler. Nunca faça INSERT ou UPDATE em tabela de outro domínio.

05 · IoT

Sensor de temperatura das vacinas.

Vacinas perdem eficácia fora da faixa 2–8°C (Anvisa RDC 197/2017). Um sensor ESP32 monitora a geladeira da clínica e envia leituras a cada 60 segundos. O banco armazena o histórico e os alertas.

DISCIPLINA: Disruptive IoT/IA 3 tabelas · Domínio .NET · Frontend clínica consome para exibir dashboard de temperatura.
DISPOSITIVO_IOT Cadastro do sensor. Cada geladeira = 1 dispositivo. DS_IDENTIFICADOR = MAC address do ESP32. IoT
ColunaTipoO que significa
DS_IDENTIFICADORVARCHAR2(60) · UNIQUEMAC address ou ID único do ESP32. O device autentica usando esse campo para enviar leituras.
NR_TEMP_MINIMA / MAXIMANUMBER(4,2)Faixa aceitável configurável por dispositivo. Default Anvisa: 2.0°C a 8.0°C.
NR_INTERVALO_LEITURANUMBER(4)Segundos entre leituras. Default 60s = 1.440 leituras/dia. Ajustável.
DT_ULTIMA_LEITURATIMESTAMPAtualizado a cada POST de leitura. Se diferença com SYSTIMESTAMP > 5× intervalo, gerar alerta SENSOR_OFFLINE.
LEITURA_TEMPERATURA Time-series. ~1.440 linhas/dia/dispositivo. Nunca deletar — fonte de auditoria de lotes. IoT
ColunaTipoO que significa
NR_TEMPERATURANUMBER(5,2)Temperatura em °C. CHECK: entre -40 e 80 (range físico do sensor).
NR_UMIDADENUMBER(5,2) · NULLSe sensor DHT11/DHT22, envia umidade também. DS18B20 não tem — deixa NULL.
ST_DENTRO_FAIXACHAR(1)Calculado pelo backend no momento do INSERT comparando com DISPOSITIVO_IOT.NR_TEMP_MINIMA/MAXIMA.
🔗 Índice: (ID_DISPOSITIVO, DT_LEITURA DESC) — query de gráfico busca por device + período
ALERTA_TEMPERATURA 1 alerta por evento problemático, não 1 por leitura. Evita spam no dashboard. IoT
ColunaTipoO que significa
DS_SEVERIDADEenumBAIXA (8-10°C) → MEDIA (10-12°C) → ALTA (>12°C) → CRITICA (>20°C ou offline >30min).
DT_FIMTIMESTAMP · NULLNULL = alerta ativo. Preenchido quando temperatura volta à faixa. Dashboard filtra DT_FIM IS NULL.
ST_RESOLVIDOCHAR(1)Resolvido manualmente (responsável identificado, ação tomada). Diferente de DT_FIM — pode ter voltado à faixa mas o alerta ainda precisa de ação humana.
DS_ACAO_TOMADAVARCHAR2(500)O que foi feito quando resolvido. Importante para auditoria de lotes junto à Anvisa.

07 · Frontend — App Tutor (Kura)

O que Guilherme precisa saber para construir as telas.

O app Kura é a interface do tutor. Você consome a API Java (Nikolas) pra dados de identidade e agendamento, e a API .NET (Felipe) pra dados clínicos do pet. As Views do banco já fazem o JOIN pesado — você recebe o dado pronto.

Autenticação JWT — como funciona

Ao fazer login, a API Java retorna um token JWT. Esse token vai no header de todas as requisições subsequentes: Authorization: Bearer {token}. O token tem expiração — implemente refresh token ou redirecionar para login. Dados do tutor logado (ID_TUTOR, NM_TUTOR) estão no payload do JWT — não precisa de GET separado pra isso.

🔔
Alertas do pet

Vem da API .NET: vacinas vencendo próximos 30 dias via VW_VACINAS_VENCENDO. Badge no card do pet. Tutor vê sem precisar entrar no prontuário.

AGENDAMENTO WHERE ID_TUTOR=? AND DT_AGENDAMENTO > SYSTIMESTAMP AND ST_STATUS IN ('AGENDADO','CONFIRMADO') ORDER BY DT_AGENDAMENTO ASC FETCH FIRST 1. Exibir abaixo do card do pet.

🏥
Clínica vinculada

Exibir NM_CLINICA e DS_TELEFONE da clínica que cadastrou o tutor. Vem pelo TUTOR.ID_CLINICA → JOIN CLINICA.

Tela de Perfil do Pet — Timeline

VW_TIMELINE_PET — use essa view

A view já faz todos os JOINs: PET + EVENTO_CLINICO + VETERINARIO + TIPO_EVENTO. Você recebe {ID_EVENTO, NM_TIPO_EVENTO, DT_EVENTO, NM_VETERINARIO, DS_DIAGNOSTICO, DS_MODALIDADE}. Ordene por DT_EVENTO DESC. Use NM_TIPO_EVENTO para escolher o ícone de cada item da timeline.

Tela de Consentimentos LGPD

Como exibir e atualizar consentimentos

Para cada tipo (LEMBRETES, TELEORIENTACAO, DADOS_ANONIMOS...), buscar o registro mais recente em CONSENTIMENTO e verificar se ST_ACEITO='S'. Exibir toggle: ligado = aceito, desligado = revogado. Ao mudar: POST para /api/consentimentos com o tipo e o novo estado. A API cria novo registro — não atualiza. O toggle representa o estado atual, não o histórico.

Tela de Receitas Ativas

Como montar a tela de prescrições

Buscar PRESCRICAO WHERE ID_EVENTO IN (eventos do pet). Para cada prescrição, buscar PRESCRICAO_MEDICAMENTO JOIN MEDICAMENTO. Montar card: nome do medicamento (NM_MEDICAMENTO), dosagem (DS_DOSAGEM), frequência (DS_FREQUENCIA), duração (NR_DURACAO_DIAS dias), instruções (DS_INSTRUCOES). Exibir data de validade da receita (PRESCRICAO.DT_VALIDADE) — destacar se vencida.

Duas views no banco fazem os JOINs pesados por você. Ambos os backends e o frontend podem consumir via endpoint da API sem montar SQL complexo.

VW_TIMELINE_PET
Retorna toda a timeline de um pet com JOIN em EVENTO_CLINICO, VETERINARIO e TIPO_EVENTO. Filtrar por ID_PET e ordenar por DT_EVENTO DESC. Usada pelo frontend tutor (tela de perfil) e pela Luna (quando tutor pergunta sobre histórico).
VW_VACINAS_VENCENDO
Retorna todos os pets com vacinas vencendo nos próximos 30 dias. Já faz JOIN com TUTOR (via TUTOR_PET onde ST_PRINCIPAL='S'), PET, VACINA e CLINICA. Retorna DS_WHATSAPP do tutor principal e DIAS_RESTANTES calculado. Job da Luna lê essa view diariamente para gerar NOTIFICACAO.

10 · Decisões LGPD

Por que podemos usar dados do tutor antes do portal.

Pergunta legítima e importante. A resposta está na base legal, não no consentimento.

✅ O que pode sem consentimento do portal
  • → Armazenar nome, CPF, telefone do tutor
  • → Manter prontuário do pet (obrigação CFMV)
  • → Emitir nota fiscal do atendimento
  • → Entrar em contato em caso de urgência
  • → Enviar lembrete de vacina (relação contratual)
Base: art. 7º, V (execução de contrato) + VI (obrigação legal)
⚠️ O que requer consentimento do portal (Java)
  • → Teleorientação via canais digitais (CFMV 1.465/2022)
  • → Comunicação de marketing
  • → Uso de dados para pesquisa/analytics
  • → Compartilhamento com seguradora
  • → Notificações promocionais
Tabela CONSENTIMENTO · Domínio Java
O campo TUTOR.ST_AVISO_PRIVACIDADE

Adicionamos esse campo na v2 do schema. Quando a recepcionista cadastra o tutor, deve marcar ST_AVISO_PRIVACIDADE='S' e DS_VERSAO_AVISO='v1.0'. Isso registra que o tutor foi informado sobre o tratamento de dados no balcão (transparência LGPD art. 6º, VI). Não é consentimento formal — é evidência de transparência. Diferença importante para eventual auditoria.
